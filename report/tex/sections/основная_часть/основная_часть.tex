\section{Основная часть}

\subsection{Подготовка виртуальной среды}

Практическая часть работы выполнялась на локальной машине с использованием Oracle VM VirtualBox\footnote{Руководство пользователя Oracle VM VirtualBox: \url{https://www.virtualbox.org/manual/}.}. В качестве модели вычислительного кластера была использована схема из пяти виртуальных машин: один управляющий узел и четыре вычислительных узла. Управляющий узел получил имя \mintinline{text}{mgmt}, вычислительные узлы получили имена \mintinline{text}{cn01}, \mintinline{text}{cn02}, \mintinline{text}{cn03} и \mintinline{text}{cn04}.

На все виртуальные машины был установлен дистрибутив openSUSE Leap\footnote{Документация openSUSE Leap: \url{https://doc.opensuse.org/}.}. При установке была выбрана минимальная консольная конфигурация без графической среды.

Каждой виртуальной машине был выделен один виртуальный процессор и 2 ГБ оперативной памяти. Первоначально для системного диска рассматривался объём 8--10 ГБ, однако установщик openSUSE Leap требовал не менее 12,5 ГиБ под корневой раздел и дополнительно 1--2 ГиБ под раздел подкачки. После этого виртуальные машины были пересозданы с диском объёмом 16 ГБ.

В результате была подготовлена однородная виртуальная среда: все узлы используют один дистрибутив, одинаковые аппаратные ограничения и минимальный набор служб. Сначала установка и базовая настройка были выполнены на управляющем узле, после чего тот же порядок действий был повторён на вычислительных узлах.

\subsection{Настройка сети и доступа}

Для каждой виртуальной машины использовались два сетевых адаптера. Первый адаптер был оставлен в режиме NAT\footnote{NAT~--- Network Address Translation, механизм трансляции сетевых адресов.}. Он нужен для доступа к внешним репозиториям openSUSE и установки пакетов через \mintinline{text}{zypper}\footnote{\mintinline{text}{zypper}~--- консольный менеджер пакетов openSUSE; справка по использованию: \url{https://en.opensuse.org/SDB:Zypper_usage}.}. Второй адаптер был подключён к внутренней сети VirtualBox. Именно через этот адаптер организована локальная связь узлов кластера между собой.

Для внутренней сети был выбран диапазон IP-адресов \mintinline{text}{10.1.0.0/24}. Управляющему узлу назначен адрес \mintinline{text}{10.1.0.21}, вычислительным узлам назначены адреса \mintinline{text}{10.1.0.31}, \mintinline{text}{10.1.0.32}, \mintinline{text}{10.1.0.33} и \mintinline{text}{10.1.0.34}. Адреса управляющего и вычислительных узлов разнесены по разным десяткам, поэтому конфигурация остаётся читаемой и допускает добавление новых узлов без изменения выбранной схемы.

Для доступа к виртуальным машинам с хостовой системы в VirtualBox были настроены пробросы SSH-портов через NAT: для \mintinline{text}{mgmt} использовался порт \mintinline{text}{2201}, для \mintinline{text}{cn01}--\mintinline{text}{cn04}~--- порты \mintinline{text}{2202}--\mintinline{text}{2205}. Это позволяло подключаться к каждому узлу с локальной машины, не открывая внешний доступ к внутренней сети кластера.

После настройки проброса портов на локальной машине была создана SSH-пара ключей и подготовлен пользовательский конфигурационный файл \mintinline{text}{~/.ssh/config}. В нём для каждого узла были указаны локальный адрес, соответствующий проброшенный порт и приватный ключ. После этого подключение с хостовой системы выполнялось по коротким именам узлов.

Разрешение имён выполнено через файл \mintinline{text}{/etc/hosts}. На всех узлах были добавлены одинаковые записи для имён \mintinline{text}{mgmt}, \mintinline{text}{cn01}, \mintinline{text}{cn02}, \mintinline{text}{cn03} и \mintinline{text}{cn04}.

Далее был настроен SSH-доступ без ввода пароля. Для этого на каждом узле была создана SSH-пара ключей, публичные ключи всех узлов были добавлены в файл \mintinline{text}{/etc/authorized/keys} на каждом узле. Дополнительно был создан пользовательский SSH-конфиг, в котором имена узлов сопоставлены с соответствующими хостами и выбранным приватным ключом. После настройки команда вида \mintinline{text}{ssh cn01} выполняется с любого узла без указания ключа и без ввода пароля.

\subsection{Установка MUNGE и SLURM}

Для работы SLURM требуется единый механизм аутентификации сообщений между узлами. Эту функцию выполняет MUNGE. На управляющем узле был создан общий ключ MUNGE, после чего он был скопирован на все вычислительные узлы. Для ключа были выставлены ограниченные права доступа и владелец \mintinline{text}{munge}; одинаковый ключ на всех узлах является условием корректного обмена служебными сообщениями SLURM.

Для корректной работы MUNGE также требуется согласованное системное время на всех узлах. При первоначальной установке на вычислительных узлах остался часовой пояс UTC+2, а на управляющем узле был установлен UTC+3. Из-за расхождения времени вычислительные узлы не могли корректно подключиться к кластеру: MUNGE проверяет время создания служебных сообщений и отклоняет сообщения, которые заметно отличаются от локального времени узла. После установки одинакового часового пояса и проверки текущего времени на всех виртуальных машинах службы MUNGE и SLURM начали взаимодействовать корректно.

Пакеты устанавливались через пакетный менеджер \mintinline{text}{zypper}. На управляющем узле используется служба \mintinline{text}{slurmctld}, которая хранит состояние очереди, принимает задания и назначает ресурсы. На вычислительных узлах используется служба \mintinline{text}{slurmd}, которая сообщает управляющему узлу о состоянии узла и выполняет назначенные задания.

Конфигурация SLURM была сформирована через официальный конфигуратор SchedMD\footnote{Slurm Configuration Tool на сайте SchedMD: \url{https://slurm.schedmd.com/configurator.html}.}. В конфигурации указан кластер \mintinline{text}{practice}, управляющий узел \mintinline{text}{mgmt}, вычислительные узлы \mintinline{text}{cn[01-04]} и очередь \mintinline{text}{debug}. Для выбора процесса отслеживания был оставлен вариант \mintinline{text}{proctrack/cgroup}; для выбора ресурсов использован \mintinline{text}{select/cons_tres}; для планировщика выбран \mintinline{text}{sched/backfill}. Эти параметры соответствуют рекомендациям конфигуратора.

Один и тот же файл \mintinline{text}{slurm.conf} был размещён на всех узлах в каталоге \mintinline{text}{/etc/slurm}. На управляющем узле был создан каталог сохранения состояния контроллера, на вычислительных узлах был создан каталог для состояния \mintinline{text}{slurmd}. После запуска служб состояние кластера проверялось командами \mintinline{text}{sinfo} и \mintinline{text}{squeue}; перед запуском генерации вычислительные узлы должны находиться в состоянии \mintinline{text}{idle}.

Команда \mintinline{text}{sinfo} показывает состояние разделов SLURM и вычислительных узлов. Пример вывода для настроенного кластера:

\begin{minted}[fontsize=\small]{text}
PARTITION AVAIL  TIMELIMIT  NODES  STATE NODELIST
debug*       up   infinite      4   idle cn[01-04]
\end{minted}

Команда \mintinline{text}{squeue} используется для просмотра очереди заданий; перед запуском генератора её пустой вывод означает, что в очереди нет активных или ожидающих заданий.

\subsection{Обработка исходного набора данных}

Индивидуальный вариант определяет три параметра: дистрибутив openSUSE Leap, UID 50728 и имя задания \mintinline{text}{qe7}. Исходный набор данных был распакован и отфильтрован по UID и имени задания. В результате сформирован файл \mintinline{text}{acct_0921-0923_uid50728_qe7}, содержащий 1563 записи.

В выбранном фрагменте 1542 записи имеют состояние \mintinline{text}{COMPLETED}, 11 записей имеют состояние \mintinline{text}{FAILED}, 10 записей имеют состояние \mintinline{text}{CANCELLED}. Для построения распределений использовались завершённые задания, так как только для них поле длительности выполнения можно интерпретировать как корректное время работы задачи.

Исходная длительность выполнения задания берётся из поля \mintinline{text}{ElapsedRaw}. В выгрузке SLURM поле \mintinline{text}{TimelimitRaw} задано в минутах, а \mintinline{text}{ElapsedRaw}~--- в секундах, поэтому при вычислении ошибки прогноза запрошенное время переводится в секунды:

\[
    \Delta T = \text{TimelimitRaw} \cdot 60 - \text{ElapsedRaw}.
\]

В выбранном наборе данных минимальная длительность завершённого задания составляет 699 секунд, медиана составляет 1652 секунды, среднее значение составляет 1712.39 секунды. Максимальное значение равно 16660 секунд. Ошибка прогноза времени для завершённых заданий имеет медиану 19948 секунд и среднее значение 19887.61 секунды; следовательно, пользовательские задания в данном варианте обычно запрашивали время с большим запасом относительно длительности из исходной статистики.

На рисунке~\ref{fig:source-summary} приведена сводная визуализация отфильтрованной выборки. Она показывает, сколько записей осталось после отбора по индивидуальному варианту, какую долю составляют успешно завершённые задания. Эти данные используются дальше при подборе распределений для генератора.

\subsection{Генерация заданий}

Генератор запускается на управляющем узле \mintinline{text}{mgmt}. На вход ему передаются число заданий, масштаб длительности выполнения, масштаб планового времени и параметры распределений. Для каждого задания выбирается исходная длительность, ошибка прогноза и число требуемых узлов. Плановое время рассчитывается как сумма выбранной длительности и ошибки прогноза, после чего значения масштабируются для запуска на кластере.

Результатом работы генератора является каталог запуска с манифестом \mintinline{text}{manifest.csv} и набором файлов заданий \mintinline{text}{.slurm}. В манифест записываются имя файла задания, сгенерированные и масштабированные времена, лимит SLURM, число узлов и число задач. Каждый файл задания содержит параметры \mintinline{text}{#SBATCH}, команду \mintinline{text}{sleep} с рассчитанной длительностью и запись времени начала и окончания в общий журнал. Пример такого задания приведён в приложении A.

Масштабирование нужно для того, чтобы исходные задания длительностью в десятки минут можно было воспроизвести на кластере за ограниченное время. Масштаб длительности выполнения применяется к выбранной длительности задания и определяет аргумент команды \mintinline{text}{sleep}. Масштаб планового времени применяется к сумме выбранной длительности и ошибки прогноза; это значение записывается в манифест как плановое время и используется при расчёте ошибки прогноза. В параметре \mintinline{text}{--time} для SLURM передаётся то же плановое время, но округлённое до минут из-за формата лимита выполнения.

В манифесте для итогового запуска оба масштаба были выбраны 0.045. После масштабирования медиана времени \mintinline{text}{sleep} составила 74 секунды, среднее значение~--- 76.116 секунды, максимум~--- 89 секунд. Медиана планового времени составила 972 секунды. Так как длительность выполнения и плановое время масштабируются одинаково, соотношение между длительностью задания и запасом времени сохраняется.

\subsection{Запуск заданий и сбор фактической статистики}

После генерации задания отправляются в SLURM командой \mintinline{text}{sbatch}. Состояние очереди отслеживается через \mintinline{text}{squeue}, а сведения о завершённых заданиях сохраняются в файл \mintinline{text}{results.csv}. Для каждого задания фиксируются идентификатор SLURM, имя файла задания, состояние, узел выполнения, время ожидания в очереди, фактическое время выполнения и наблюдаемое полное время.

При анализе используются три временные величины. Запланированное время берётся из манифеста генератора. Значение \mintinline{text}{--time} передаётся в SLURM как лимит выполнения, но SLURM округляет его до минут. Фактическое время выполнения берётся из статистики SLURM. Поэтому фактическая ошибка прогноза считается по плановому значению из \mintinline{text}{manifest.csv}, а не по округлённому лимиту из очереди.

В итоговом запуске все 500 заданий завершились в состоянии \mintinline{text}{COMPLETED}. Запуск занял 6 часов 7 минут 4 секунды. Суммарное фактическое время выполнения всех заданий составило 10 часов 38 минут 28 секунд; оно больше времени всего запуска, так как задания выполнялись параллельно. Запланированное время \mintinline{text}{sleep} находилось в диапазоне от 72 до 89 секунд, медиана составила 74 секунды. Фактическое время выполнения находилось в диапазоне от 72 до 90 секунд, медиана составила 75 секунд. Медиана накладных расходов относительно команды \mintinline{text}{sleep} составила 0.562\%, среднее значение~--- 0.656\%, максимальное значение~--- 1.389\%.

На рисунке~\ref{fig:planned-actual} показано сравнение планового и фактического времени по отдельным заданиям.

\screenshot[fig:planned-actual][1]{img/planned_vs_actual_by_job.png}{Плановое и фактическое время выполнения заданий}

Отклонение фактического времени по каждому заданию считается относительно времени \mintinline{text}{sleep}, записанного генератором в файл задания:

\[
    \delta_i =
    \frac{
        T_{\text{run}, i}^{\text{SLURM}} -
        T_{\text{sleep}, i}^{\text{manifest}}
    }{
        T_{\text{sleep}, i}^{\text{manifest}}
    }
    \cdot 100\%.
\]

Здесь $T_{\text{sleep}, i}^{\text{manifest}}$~--- запланированная длительность команды \mintinline{text}{sleep} для $i$-го задания, а $T_{\text{run}, i}^{\text{SLURM}}$~--- фактическое время выполнения этого задания по статистике SLURM. На рисунке~\ref{fig:actual-overhead} показаны значения $\delta_i$ по всем заданиям. Они отражают накладные расходы запуска и завершения заданий в SLURM и виртуальной среде. В итоговом запуске все значения остаются ниже порога 3\%.

\screenshot[fig:actual-overhead][1]{img/actual_overhead_by_job.png}{Накладные расходы фактического выполнения по заданиям}

Для сравнения был рассмотрен более короткий предварительный запуск на 200 заданий с масштабом времени 0.01, то есть с ускорением в 100 раз. В этом случае длительность \mintinline{text}{sleep} для большинства заданий попадает в диапазон 10--20 секунд, поэтому постоянные накладные расходы SLURM занимают заметную долю времени выполнения.

\screenshot[fig:actual-overhead-200-100x][1]{img/actual_overhead_200_values_100x.png}{Накладные расходы при запуске 200 заданий с ускорением в 100 раз}

На рисунке~\ref{fig:actual-overhead-200-100x} часть значений выходит за контрольный порог 3\%. В этом запуске порог превысили 139 из 200 заданий, максимальное отклонение составило 12.5\%, медианное значение~--- 5.0\%. Поэтому для итогового запуска был выбран менее агрессивный масштаб времени: он увеличивает длительность отдельных заданий, но уменьшает относительный вклад накладных расходов.

\subsection{Анализ фактической ошибки прогноза}\label{sec:actual-forecast-error}

Для удобства сопоставления на рисунке~\ref{fig:error-fit-actual-comparison} повторно приведено распределение ошибки прогноза времени с наложением фактической ошибки после запуска заданий.

\screenshot[fig:error-fit-actual-comparison][1]{img/forecast_error_fit.png}{Сопоставление расчётной и фактической ошибки прогноза времени}

Голубая гистограмма показывает запас времени, заложенный генератором до отправки заданий в очередь, оранжевая гистограмма показывает тот же запас после замены времени \mintinline{text}{sleep} на фактическое время выполнения из статистики SLURM. Распределения лежат в одном диапазоне, поэтому накладные расходы запуска не меняют характер ошибки прогноза, а только немного сдвигают отдельные значения.

После получения статистики SLURM ошибка прогноза пересчитывается для фактически выполненных заданий. Она определяется как разность между масштабированным плановым временем из манифеста и фактическим временем выполнения:

\[
    \Delta T_{\text{fact}} =
    T_{\text{plan}}^{\text{manifest}} -
    T_{\text{run}}^{\text{SLURM}}.
\]

Здесь $T_{\text{plan}}^{\text{manifest}}$~--- плановое время из манифеста генератора, а $T_{\text{run}}^{\text{SLURM}}$~--- фактическое время выполнения, полученное из статистики SLURM.

На рисунке~\ref{fig:actual-error-fit} показано распределение фактической ошибки прогноза. В запуске на 500 заданий ошибка находилась в диапазоне от 881 до 901 секунды, медиана составила 897 секунд. Наложенная кривая описывает наблюдаемую фактическую ошибку после выполнения заданий, а не параметры генератора.

\screenshot[fig:actual-error-fit][1]{img/actual_error_fit.png}{Распределение фактической ошибки прогноза времени}

На рисунке~\ref{fig:log-actual-error-fit} показан десятичный логарифм фактической ошибки прогноза. Фактические значения образуют компактную область рядом с распределением, заданным параметрами генератора. Небольшой сдвиг связан с тем, что фактическое время выполнения немного отличается от запланированного времени \mintinline{text}{sleep}.

\screenshot[fig:log-actual-error-fit][1]{img/log_actual_error_fit.png}{Распределение логарифма фактической ошибки прогноза времени}

Итоговый запуск на 500 заданий показал, что выбранный масштаб времени подходит для проверки на четырёх вычислительных узлах. Медианное отклонение фактического времени от \mintinline{text}{sleep} составило 0.562\%, среднее значение~--- 0.656\%, максимальное значение~--- 1.389\%. Фактическая ошибка прогноза остаётся близкой к расчётной ошибке из манифеста; это сопоставление показано на рисунке~\ref{fig:error-fit-actual-comparison}.
